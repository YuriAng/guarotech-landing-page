# design-sync notes — guarotech-landing-page

## Repo shape

- **Storybook shape.** `.storybook/` lives at repo root (`storybookConfigDir: ".storybook"`).
- **No separate component-library build.** This repo ships an app (`vite build` → `index.html`/`confirm.html`), not a publishable package. The converter's `--entry` points directly at `src/index.ts` (a re-export barrel created solely for design-sync — not imported by the app itself) and esbuild-bundles straight from TSX source. `cfg.buildCmd: "npm run build:types"` is NOT a JS/dist build — it only regenerates the `.d.ts` tree the converter's ts-morph pass needs (see next point). Re-run it whenever component prop types change; re-run `npx storybook build -c .storybook -o .design-sync/sb-reference` whenever `styles.css`, `src/assets/**`, or any component/story changes.
- **`.d.ts` generation is mandatory — `[GENERAL]`.** The repo's real `tsconfig.json` has `noEmit: true` (it's an app, never emits declarations), but `package-build.mjs`'s component/prop discovery is entirely ts-morph/`.d.ts`-driven — it does NOT read TSX source directly. Without emitted `.d.ts`, the build silently finds **0 components** (`exported PascalCase symbols: 0`, all 8 titles land in `[TITLE_UNMAPPED]`) even though the bundle itself contains all the exports. Fix: added `tsconfig.types.json` (declaration-only, `outDir: dist-types`, includes `src/vite-env.d.ts` for the Vite asset-import ambient types) + `npm run build:types` + `"types": "dist-types/index.d.ts"` in `package.json` so `findTypesRoot` finds it. **Re-sync risk:** if a future sync forgets to run `build:types` after editing component props, the stale `dist-types/` silently describes the OLD prop shape — always re-run it before the converter build, same as any other `buildCmd`.
- **Images live in `src/assets/`, not `public/assets/` — `[GENERAL]`, structural.** Originally all images lived in `public/assets/` and were referenced by absolute URL (`/assets/Logo.png`), which only resolves on the real deployed site (served from its domain root) — every `<img>` and the one CSS `background-image` 404'd inside the isolated design-sync bundle (confirmed via broken-image icons in Hero, TechSlider, ProcessSlider captures). Fixed by moving `public/assets/` → `src/assets/` and converting every reference to a real ES import (`import logoUrl from '../../assets/Logo.png'`), so esbuild inlines them as data URIs into the bundle (Vite does the equivalent hashed-file treatment for the real app — verified no regression in either). The CSS `background-image: url(/assets/Vector.svg)` (`.footer-image`) was inlined as a base64 `data:image/svg+xml;base64,...` URI directly in `styles.css` instead, since the converter's main-bundle esbuild loader map has no generic mechanism for resolving CSS `url()` references to copied files. One `.jpg` (`planificacion.jpg`) had to be converted to `.png` — the converter's default esbuild loader map (`bundle.mjs`, not configurable via `cfg`) only maps `.svg`/`.png`/`.woff`/`.woff2` to `dataurl`, not `.jpg`; `UI.jpg` was an unused duplicate and was deleted rather than converted. **Re-sync risk:** any NEW image asset added to a component must be a `.png`/`.svg`/`.woff(2)` (not `.jpg`/`.jpeg`) and imported via a relative ES import from `src/assets/`, never referenced as an absolute `/assets/...` string — the latter will silently 404 in the uploaded bundle even though it works fine in the real app and in `npm run dev`.
- **`cfg.overrides.<Name>.viewport` for tall components — `[GENERAL]`.** The converter's per-story preview capture (`storyShot()` in `compare.mjs`, and the same mechanism the app's card renderer uses) takes a fixed-viewport screenshot (`fullPage: false`, default 900×700), NOT an element-bounds capture — unlike storybook's own reference capture, which screenshots the actual DOM element regardless of height. Any component taller than 700px silently gets cropped in previews/captures with no error. Fixed via `cfg.overrides.<Name>.viewport: "WxH"` for `Hero` (`900x1120`) and `ContactForm` (`900x1450`), sized from the storybook reference's own natural element height (checked with `sips -g pixelHeight`). **Re-sync risk:** if `Hero` or `ContactForm` content grows taller, or a new tall component is added, check its raw capture for a cropped bottom edge and bump/add a `viewport` override — validate/compare won't warn about this on their own.

## Fixes applied

- **`[GENERAL]` Missing `--color-white` CSS custom property.** `src/styles/styles.css` referenced `var(--color-white)` in `.btn-ghost` and `.btn-secondary:hover` but never defined it in `:root` — a latent bug in the original static site, invisible there because neither class was ever actually used in the shipped markup (only `btn-primary`/`btn-gradient` are). Exposing `Button` variant="ghost" via Storybook surfaced it. Fixed by adding `--color-white: #ffffff;` to `:root` — matches the token's evident intent and all 3 usages. Zero effect on the real site (dead code path there).

## Accepted / known warnings (do not re-litigate on re-sync)

- **`[FONT_MISSING]` "Inter", "JetBrains Mono".** The original static site never shipped these fonts either — no `@font-face`, no Google Fonts `<link>`, anywhere in the source repo (verified against the pre-refactor git snapshot). Accepting system-font substitutes is faithful to the original, not a regression.
- **Button "Gradient" story renders full-width in the preview, compact in storybook — `close`, not fixable.** Storybook's `layout: 'centered'` story parameter isn't replicated by the converter's preview harness (confirmed: no `layout`/`centered` handling anywhere in `.ds-sync/lib` or `.ds-sync/storybook`), so the isolated single-story preview shows `.btn-gradient`'s true unconstrained default (block-level, `display:flex` → fills its container) while storybook's decorator artificially centers/shrinks it. Not a defect: real usage always supplies a resolving container — compact inside a flex parent (Hero's `.hero-cta`), intentionally full-width inside a block parent (ConfirmMessage's `.confirm`, a deliberately wide CTA). The `AllVariants` story (own flex-row wrapper) proves the component itself is fine.
- **Translucent/white-on-transparent styling reads wrong on the white product-card canvas — `close`, not fixable, affects `Button` (Ghost), `Header`, `TechSlider`.** This design system is dark-theme-only: `.btn-ghost` (transparent bg + white text), `.header-nav` and `.aliados` (both `background: var(--color-surface-2)`, a 53%-alpha dark grey) all assume a dark page behind them. The generated preview card hardcodes `body{background:#fff}` for a neutral picker canvas, so these read as invisible/washed-out grey there even though they're pixel-correct in storybook (dark iframe) and the real site. Documented in `.design-sync/conventions.md` (the "dark-theme only" note) so the design agent knows to always wrap compositions in a dark surface. Not fixable without changing the source design system's actual styling (out of scope for a sync).
- **`TechSlider`'s 6th logo nearly clipped at the right edge — `close`, not fixable.** `.slider-track` uses `width:100vw` / `justify-content:space-between`, which resolves against the real browser viewport in the isolated capture rather than the card's own container width, slightly overflowing. Cosmetic, verified correct in real usage/storybook.
- **`Footer` is genuinely broken in isolation — graded `mismatch`, will always show as `pendingGrade`, deliberately NOT skipped.** `.footer{position:absolute;bottom:0}` is designed to anchor against the real page's full document height (the real site's `.gradient-background` section reserves large bottom padding specifically to make room for the footer overlapping it). In any isolated single-component preview there's no tall page content above it to push the anchor point into view, so almost all footer content (logo, contact info, wave background) renders off-screen — only a sliver of the copyright bar is visible. Two `cardMode` options were tried: `"column"` (current) partially crops it near the top; `"single"` is worse — its wrapper's `transform:translateZ(0)` establishes a new containing block for `position:absolute` descendants, and since Footer's only child is itself out-of-flow, the wrapper collapses to zero height and pushes the bottom-anchored footer fully off-screen (`[RENDER_BLANK]`). Reverted to `"column"`. **Deliberately not using `cfg.overrides.Footer.skip`**: skip removes the story's cell from the product card entirely (confirmed in `emit.mjs` — it filters `visibleStoryIds`, which drives card generation, not just grading), and Footer only has one story — skipping it would leave a fully empty card, which is worse than the current partially-visible one. This means `Footer` will perpetually show in `verification.pendingGrade` on every future compare/resync run (a `mismatch` grade is never treated as a stable/carried-forward verdict) — this is expected, not a regression to chase. Verified correct in real usage (dev server) and storybook (whose canvas/docs wrapper happens to provide enough height to anchor it correctly).

## Re-sync risks

- Forgetting `npm run build:types` before a re-sync build silently produces 0 discovered components (see above) — it doesn't error, it just reports `[TITLE_UNMAPPED]` for everything. Always run it first.
- New image assets must go in `src/assets/` (imported via relative ES import) in `.png`/`.svg`/`.woff(2)` format — never `public/assets/` or `.jpg`. See "Images live in `src/assets/`" above.
- `cfg.overrides` (`ConfirmMessage`/`Footer`/`TechSlider`: `cardMode: "column"`; `Header`: `cardMode: "single", primaryStory: "Default"`; `Hero`/`ContactForm`: `viewport` overrides) were set from validator/capture findings on the first sync — if new stories are added to these components, re-check validate output for fresh `[GRID_OVERFLOW]` flags and re-check tall components' raw captures for cropping.
- `Footer` will always appear in the driver's `verification.pendingGrade` — this is a known, accepted, structural limitation (see above), not a sign something broke.
- If the source design system ever gains a light/white section, the "translucent/white styling needs a dark backdrop" acceptance above should be re-examined — it was accepted specifically because the whole DS is dark-only today.
- `.design-sync/conventions.md` is hand-authored (do not regenerate) — it documents the dark-theme-only requirement and the real token/class vocabulary. Keep it in sync if new tokens or button variants are added.
